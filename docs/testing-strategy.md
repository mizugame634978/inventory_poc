# テスト戦略の総括

> このプロジェクトのテスト構成と、その背後の判断・限界・製品化時の指針をまとめる。
> 方針は一貫して **「ロジックを安く厚く・UIは薄く健全性だけ・境界は契約で押さえる」**(自走に効く)。(2026-06-21)

| プロジェクト/ファイル | 層(守る対象) | 件数 | 速度 | ヘッドレス | 高速ループ(check.ps1) |
|---|---|---|---|---|---|
| server `test_domain.py` | ドメイン単体(不変条件) | (server計50) | 即 | ◎ | ✅ |

## 効いた設計
1. **ロジックをテストしやすい層へ寄せた**(ドメイン=純Python、ViewModel=WPF非依存)→挙動の大半が速い・ヘッドレス・決定的に覆える=自走の土台。
2. **テストダブルをインターフェース/DIで**(server=Depends差し替え+InMemory、client=IProductApiClientモック)。
3. **テスト間の独立**(各テスト新規InMemory、SQLite/契約は一時ファイル/ユニークSKU)。
4. **不変条件＝テスト**(在庫非負・二重入荷禁止…)=実行可能な仕様。
5. **高速ループと重いテストを分離**。check.ps1 は決定的・ヘッドレスのみ。UI(デスクトップ依存)・契約(python/uv+サーバ起動)は自走ループ外=手動/リリースゲート。
6. **「効力」を検証**(UIテストはわざと壊して検出を確認/契約テストはサーバ未起動でfixture例外)。
7. **契約境界を埋めた**(モックでは捕まえないシリアライズ/契約ズレを実サーバ契約テストで補完)。

## 正直な穴・限界(意図的なトレードオフ)
- クリック→コマンド→描画の**UI自動E2Eは無い**(FlaUI不採用)。実クリック経路は手動確認頼り。
- **UIテストはデスクトップ必須**でヘッドレス不可→自走ループ外。
- **契約テストも高速ループ外**。check.ps1 だけだと契約ズレがリリース手動実行まで滑り得る。
- **「テスト赤→ループ安全停止」は実地未観測**(機構はある)。
- **カバレッジ計測なし**(意図的に%でなく意味あるテストでの挙動網羅を優先)。
- **テストを書いた本人が実装**=自己整合の懸念(issueに受け入れテストを先に書いて緩和)。

## 製品化で足すもの
- 重要動線だけの **UIスモークE2E**(WPFなら FlaUI)。
- **契約テストをCIの必須ゲート**に(手動依存をやめる)。
- **カバレッジ計測**(coverlet)を可視化(未テスト領域の発見用途)。
- ヘッドレスCIなら **UIテストのShow依存の扱い**を設計(下記)。

### 深掘り: FlaUI スモークE2E 追加の実像(アーキ変更なし?)
ほぼ追加だけだが、アプリ側に小さなテスト容易化が要る(構造は不変):
- 追加: `InventoryClient.E2ETests`(net10.0-windows)+ `FlaUI.UIA3`。フィクスチャで**サーバ(subprocess)とWPF実exeの両方を起動**して実操作。
- アプリ小改修: ①**接続先URLを設定可能に**(今は MainWindow がハードコード→環境変数/引数)、②**AutomationProperties.AutomationId 付与**(UIAで要素を安定取得)。
- 不変: Core/WPF/Tests分割・MVVM・domain/repo(VM/ドメインは無改修)。

### 深掘り: ヘッドレスCIでの Show() 依存の扱い(比較→結論)
| 手段 | メリット | デメリット |
|---|---|---|
| A. hosted Windows runner で全部一括 | 追加インフラ不要・手軽。通常WPF描画/UIAは動く | 真のヘッドレスでなくUIA一部がフレーキー化し得る。PRが遅く/不安定化 |
| B. self-hosted + 自動ログオン対話セッション | UI自動化に最も堅牢 | マシン維持コスト・自動ログオンのセキュリティリスク・運用負荷 |
| C. ステージ分離(hosted) | PRは速く堅い・UIフレーキーがマージを止めない・既存方針と一致 | UIデグレ検知がPR時点でなく後段(夜間等) |
| D. 仮想ディスプレイ等 | — | WPFでは実質不可(問題はGPUでなくセッション) |

**結論(本番でスムーズに)= C を hosted Windows runner で**:
- PR必須ゲート = check.ps1相当(server pytest + ViewModel)+ 契約テスト。
- 別ジョブ(非ブロッキング/夜間)= UIバインディング + FlaUI スモークE2E を hosted Windows で。
- 理由: hosted で Show()/UIA は通常動くので自前インフラ(B)の負担を避けられ、UIを隔離してフレーキーがマージを止めない。**エスカレーション**: hostedで実際にフレーキーなら、その時だけUIジョブを B(自前+自動ログオン)へ格上げ。最初から重いBを背負わない。
