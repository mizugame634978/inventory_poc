# UI 自動テストの比較(WPF vs React)と課題整理

> 「軽く自動テストしたい」という狙いに対し、WPF の GUI テストがどの程度重いのかを、
> React(Web)と明示的に比較して把握するための文書。比較から見えた WPF の課題と、
> 改善の方向性の種をまとめる。(issue #0005 / 2026-06-21)

## 1. 背景・なぜ比較したか
このプロジェクトは「Claude が自律的に・人手を減らして開発する」研究が主目的(→ `CLAUDE.md`)。
そのため自動テストは **速く・決定的・無人で回せる** ことが重要。ところが WPF では、
ViewModel のロジックは軽くテストできても(`InventoryClient.Tests`)、**UI のデグレ(XAMLバインディング切れ)
を捕まえるには実描画 `Show()` が必要**で、対話デスクトップ依存になることが #0004 で分かった。
「軽く UI も自動テストしたい」という当初の期待と、WPF の現実にギャップがある。これを言語化する。

## 2. 用語の対応(WPF ↔ React)
- WPF の **FlaUI**(別プロセスの .exe を UI Automation 経由で実操作する E2E)
  ≒ React の **Playwright / Cypress**(実ブラウザを操作する E2E)。
- 重要: React には **jsdom 層**(React Testing Library + Vitest/Jest = Node内のシミュレートDOMで
  コンポーネントを描画してイベント発火)があり、**これに綺麗に対応する WPF のレイヤーが無い**。
  WPF は「ちょっと UI を触る」だけで実描画が要るため。

## 3. 同格比較: FlaUI(WPF) vs Playwright(React)
どちらも「実描画・別プロセスでユーザーのように操作する」E2E。

| 観点 | FlaUI (WPF) | Playwright (React) |
|---|---|---|
| ヘッドレス | ✕ 実デスクトップ必須 | ◎ 標準でヘッドレス、ディスプレイ不要 |
| 待機処理 | 自前でリトライ/待ち→フレーキー | ◎ auto-waiting(操作可能まで自動待機)→安定 |
| 速度 | アプリ起動が重い(数秒/件) | 起動軽く使い回せる(数百ms〜) |
| 並列実行 | 苦手(デスクトップ/フォーカス共有) | ◎ ブラウザコンテキストで容易に並列 |
| デバッグ | 弱い | ◎ trace viewer・動画・スクショ・タイムトラベル |
| セレクタ | UIA の AutomationId を探す | getByRole / getByText など意味ベース |

根本理由: **Web は最初から自動化・ヘッドレス前提で作られている**(DOMは完全に検査可能、CDP/WebDriver BiDi
が一級市民)。一方 **WPF の UIA はアクセシビリティ用に後付け**された経路なので重く・素直でない。

## 4. もう一段の差: 重いツールを出す頻度
React は大半を **jsdom 層**で前倒し検証できる(1件あたり数〜数十ms、ほぼフレーク無し)。
そのため「FlaUI相当=実クリックE2E」を出す場面自体が少ない。

テストピラミッドの形が変わる:
- React: 大量の jsdom テスト(激軽) + 少数の Playwright E2E(中)
- WPF: in-process バインディングテスト(中・要 Show) + 必要時のみ FlaUI(重)

## 5. 自律(無人実行)との相性
| 手段 | 1件あたり(オーダー) | フレーク | 無人実行 |
|---|---|---|---|
| jsdom (React Testing Library) | ms〜数十ms | ほぼ無 | ◎ |
| Playwright (React) | 数百ms〜数秒 | 低 | ◎(ヘッドレス) |
| in-process WPF(本PJの #0004) | 約0.6秒 | 低 | △(Show=デスクトップ依存) |
| FlaUI (WPF) | 数秒〜 | 中〜高 | △〜✕(デスクトップ必須) |

## 6. 比較から見えた WPF の課題
1. **「軽い UI 自動テスト」がやりにくい。** UI に少しでも触れると実描画が要り、ヘッドレス/無人実行と相性が悪い。
2. **jsdom 相当の中間層が無い。** ロジック(ViewModel)テストと実描画テストの間が断絶していて、
   段階的に薄く検証する手段が乏しい。
3. **E2E(FlaUI)がフレーキー&デスクトップ依存。** 自走ループに組み込むと不安定要因になりやすい。
4. 結果として **テストの重心が ViewModel に偏らざるを得ない**(=View 側の安全網が薄い)。

## 7. 改善の方向性(種。すべてやるとは限らない)
- **View を極限まで薄く保つ(humble object)。** ロジックを Core/ViewModel に寄せ、View で壊れ得る面積を最小化する。
  → テストできない領域を「設計で減らす」。本PJの基本方針(→ `docs/architecture.md`)。
- **バインディングエラーをゲート化。** #0004 のバインディングエラー検出を全画面に広げ、CI/コミット前に必ず回す。
  実行時の `PresentationTraceSources` 監視をデバッグ起動時にも有効化し、開発中に即気付けるようにする。
- **FlaUI は限定導入。** 「クリック→コマンド発火」「実際に見えて押せるか」を保証したい少数の重要動線にだけ使い、
  自走ループ本体からは分離(手動/夜間バッチ等)する。
- **題材としての割り切り。** UI テストが難しいのは WPF 研究の本質的な論点。むしろ「GUIテストが難しい環境で
  どこまで自走できるか」を観察対象にする。
- (将来検討) UI 技術自体の選択肢比較。同じ業務ロジック(server/Core)を保ったまま、別フロント(例: Web)を
  載せ替えたときにテスト容易性がどう変わるかは、別途実験する価値がある。

## 8. 結論(一言)
FlaUI 相当(Playwright)単体でも、ヘッドレス・自動待機・並列・デバッグの分だけ Web の方が軽い。
さらに React は jsdom 層で重いツールの出番を減らせるため、総合的に WPF よりかなり身軽。
WPF を選んでいる本PJでは「軽い UI 自動テスト」は素直には得られない前提で、**View を薄く保つ設計** と
**バインディング健全性チェックの徹底** で実利を取りにいくのが現実解。
