  核心: スタック依存なのは「checkの中身」だけ

  ハーネスの本質は 「①単一のcheckコマンド（オラクル）＋②Stop
  hookで閉じたループ＋③CLAUDE.md/docs/issues/memoryで知識永続化＋④issue/branch/mergeワークフロー＋⑤ガードレール（テストゲート・受け入れテスト必須）」。
  このうちスタックに依存するのは①checkの“中身”だけで、残りはほぼそのまま運べます。check.ps1 の「$PSScriptRoot起点で各テストを走らせ、失敗で非0を返し集約する」骨格は不変で、uv run pytest / dotnet test
  の行を新スタックのコマンドに置き換えるだけです。

  再利用手順（推奨ステップ）

  1. 新リポジトリを git init。作業ルート＝プロジェクト直下に固定する（settings/hook/CLAUDE.mdが読まれる条件。claude-code-settings-scope.md の教訓）。
  2. 骨組みをコピー: scripts/（check/stop-check）、.claude/settings.json、CLAUDE.mdテンプレ、docs/workflow.md、docs/autonomy-harness.md、issueテンプレ。
  3. まずオラクルを通す: 新スタック用に check.ps1 を書き換え、「1コマンドで全fastテストを実行し、失敗で非0」を満たす。テストが0件でも緑になることを確認。ここが全ての土台。
  4. permissions と CLAUDE.md を新スタックに合わせる（テスト/ビルド/git コマンド、プロジェクト目的・規約）。
  5. Stop hookの発火を確認（プロジェクトから claude 起動 → /hooks に出る → わざとテストを赤にして停止がブロックされるか）。
  6. issue駆動で開発（1機能=受け入れテスト付きissue→ブランチ→check緑→--no-ffマージ）。必要なら /loop（受け入れテスト必須版）で自走。

  再利用を制限する注意点（正直に）

  - 起動ルート依存: Stop hook/settings/CLAUDE.md は「claudeを起動したディレクトリ直下」しか読まれない。新プロジェクトでもそこから起動する運用が前提。この知識ごと claude-code-settings-scope.md
  を一緒に持っていくと事故が減る。
  - OS差: 今回は PowerShell。非Windowsなら check/stop-check を bash へ移植（構造は同じ、構文だけ）＋ Stop hook を shell: bash に。
  - オラクルの質が全て: 新プロジェクトでもテストが速い・決定的・ヘッドレスでないと、Stop
  hookループが遅く/フレーキーになる（今回のUIテストと同じ轍）。だから新プロジェクトでも**「ロジックをテスト可能な層へ寄せる」原則**を最初から効かせることが、ハーネスを活かす前提条件。

  パッケージ化の案（さらに楽にするなら）

  - 最小: harness-template/ ディレクトリ（scripts＋settings骨組み＋CLAUDE/workflow/autonomy-harnessテンプレ＋6ステップのbootstrapチェックリスト）を作り、新規プロジェクトにコピーする。
  - 進んだ案: Claude Code の skill/plugin として「ハーネス雛形を生成するコマンド」にする。update-config 的に settings を組み、scripts を配置するところまで自動化できる。

  一番効いた要素（再利用時に最優先で持っていくもの）

  実験を通して効果が大きかった順:
  1. 単一の check オラクル（自走の合図・最高レバレッジ）
  2. 受け入れテスト付きの issue（ループ成否を左右する＝issueの質）
  3. 構造的ガードレール（テストゲート）＞ behavioral（「赤ならマージ不可」は堅い）
  4. 起動ルート固定の規律

  この4つを最初に据えれば、スタックが違っても同じ自走体制が立ち上がります。
